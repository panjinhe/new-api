## 无中断部署改造计划

### 目标

- 将当前“单实例原地替换”的发布方式，升级为“尽量不中断用户请求”的发布流程。
- 对普通短请求做到无感切换。
- 对流式请求 / SSE / 长连接做到尽量不被强断。

### 当前进展

- 当前应用入口已改为 `http.Server` 启动，并监听 `SIGTERM` / `SIGINT` 做优雅停机。
- 已新增蓝绿部署配置：`new-api-blue` 绑定 `127.0.0.1:3001`，`new-api-green` 绑定 `127.0.0.1:3002`。
- 已新增 `scripts/deploy-bluegreen-prod.ps1`，继续走本地构建上传路径，不在生产服务器做 Docker build。
- 已新增 Nginx upstream 模板，发布时切 upstream 并 reload Nginx。
- 已新增 active-color 机制，蓝绿实例都可承接 HTTP，但只有 active 实例运行后台维护任务。
- 已在停机路径 flush 批量用量、配额和数据看板缓存，降低旧实例 drain 后丢数据的风险。
- 已为数据库迁移加上实例间互斥锁，避免蓝绿同时启动时并发 `AutoMigrate`。
- 已在本地隔离环境用生产 PostgreSQL 快照演练过蓝绿启动、健康检查、切流和停机路径。

### 数据同步策略

- PostgreSQL 是唯一事实源，蓝绿实例共用同一个 `SQL_DSN`，不做应用到应用的数据复制。
- JWT 登录态不依赖单实例 session，切流不会踢用户下线。
- 配额、用量、请求数通过数据库原子增量和停机 flush 保证不丢。
- Redis 暂不是 v1 前置条件；如果后续做长期双活，再把分布式限流、缓存一致性和分布式锁作为 Redis 专项推进。
- 文件状态保持最小共享：`data-prod` 共用，日志按 blue / green 分目录。

### 发布流程

1. 本地构建 Linux binary。
2. 上传源码归档和 binary 到服务器。
3. 检测当前 active color，选择 idle color。
4. 启动 idle 实例并替换 `/new-api` binary。
5. 等待 idle 实例本地健康检查通过。
6. 原子切换 Nginx upstream 并 reload。
7. 公开地址 smoke test 通过后，切换 active-color 标记。
8. 旧实例按 `GRACEFUL_SHUTDOWN_TIMEOUT_SECONDS` / compose `stop_grace_period` drain 后停止。

### 已实现项

- `docker-compose.prod.bluegreen.yml`
- `scripts/deploy-bluegreen-prod.ps1`
- `ops/nginx/new-api-bluegreen-upstream.conf`
- `ops/nginx/new-api-bluegreen-proxy-snippet.conf`
- `BACKGROUND_TASKS_ENABLED` / `INSTANCE_COLOR` / `ACTIVE_COLOR_FILE` 运行时控制
- `model.FlushBatchUpdates()` 停机落库

### 后续待做

- 先用 `-DryRun` 检查发布参数和远端路径。
- 首次线上演练建议低峰执行，并实时观察 `/api/status`、Nginx 5xx、容器健康状态。
- 完成至少 3 次连续 blue / green 切换演练后，再把 `deploy-fast-prod.ps1` 标记为备用路径。
- 所有数据库迁移仍需前后版本兼容，禁止切流阶段执行破坏性迁移。

### 风险点

- 流式请求在旧实例下线时，仍可能因为等待时间不足而中断。
- 如果新旧版本共享状态但数据结构不兼容，蓝绿部署仍会失败。
- 首次从 legacy `new-api-prod` 迁移到 blue / green 时，需要确认 Nginx 站点配置能被脚本正确 patch。
- 若后续改成长时间双活，需要补 Redis 或分布式锁策略。

### 验收标准

- 发布过程中首页和 API 健康检查持续可访问。
- 普通 API 请求在发布期间不出现明显 5xx 峰值。
- 流式请求中断率显著低于当前方案。
- 可以稳定完成至少 3 次连续蓝绿切换演练。

- `lottie-web` `eval` 告警：低 ROI，暂不单独立项。

- 继续追该 warning 需要扩大前端依赖升级范围，当前用户收益有限，暂不作为近期专项。

### 当前判断依据

- 已完成控制台主要页面的路由级懒加载，`Token`、`Log`、`Setting`、`Playground`、`Channel` 等页面不再直接进入主入口包。
- 最新构建中主入口包已从约 6.86MB 降至约 4.55MB，gzip 后从约 1.61MB 降至约 0.97MB。
- 构建仍有大 chunk 警告，主要集中在 `Semi UI`、Markdown/Mermaid/KaTeX 渲染链，以及一个共享业务 chunk。
- `lottie-web` 不是业务代码直接引入，而是通过 `@douyinfe/semi-ui` 依赖链带入。

### 建议目标

- 不追求“清空所有 warning”，优先做对用户体验真正有收益的体积优化。
- 重点降低首屏和控制台首次进入时的下载体积。
- 将低价值告警并入后续依赖维护，而不是单独花时间清理。

### 推荐后续动作

1. 暂不继续拆大 chunk，除非真实用户加载瀑布图或生产访问数据证明它仍是主要瓶颈。
2. 后续如继续优化，优先对重依赖模块做按需延迟加载，关注：
   - Markdown / Mermaid / KaTeX 渲染链
   - Dashboard 图表相关依赖
   - `react-fireworks` 这类低频特效依赖
3. 对共享业务 chunk 先做构成分析，再决定是否拆分，避免为了消除 warning 增加过多请求和维护复杂度。
4. 每轮优化后重新执行 `bun run build`，记录主包和首屏相关 chunk 体积变化。
5. 在下一次前端依赖升级时顺手处理 Browserslist 数据旧告警。
6. `lottie-web` 仅在以下场景再单独处理：
   - 需要收紧 CSP
   - 安全扫描要求清理 `eval`
   - 上游 `Semi UI` 提供更干净的替代版本

### 暂定优先级

- Done：首屏 / 路由拆包优化
- P2：真实加载瀑布图分析后，再决定是否继续拆 Markdown/Mermaid 和共享业务 chunk
- P3：Browserslist 数据旧告警，等常规前端依赖升级时一并处理
- P3：`lottie-web` 告警跟踪，不主动投入专项时间

## Uptime Kuma 监控面板 ROI 计划

### 当前决策

- 暂时禁用控制台「服务可用性」面板。
- 原因是当前没有接入 Uptime Kuma 公开状态页，空卡片会降低用户观感。
- 该面板只用于展示监控状态，不影响 API 调用、渠道调度、计费或令牌功能。

### ROI 判断

- 未配置监控时展示空卡片：负 ROI，容易让用户误以为站点未维护。
- 在用户量较小时立即搭建完整监控体系：中低 ROI，维护成本和误报成本偏高。
- 有付费用户或客服咨询量上来后，接入少量核心监控：中高 ROI，可减少「是不是平台挂了」这类重复咨询。

### 低成本推荐方案

后续如需启用，优先使用独立环境部署 Uptime Kuma，并只监控 2-4 个核心公开地址：

1. `https://aheapi.com/`
2. `https://aheapi.com/api/status`
3. `https://aheapi.com/v1/models`
4. `https://aheapi.com/logo-001.jpg`（可选）

建议监控间隔设置为 60-120 秒，失败重试 2-3 次，避免网络抖动造成误报。

### 后续启用条件

- 用户开始频繁询问服务是否可用。
- 有稳定付费用户，需要提高信任感和降低售后成本。
- 已有独立机器或外部环境可部署 Uptime Kuma，避免和主站同机宕机导致监控一起失效。

### 后续配置方式

1. 在 Uptime Kuma 中创建公开 Status Page，例如 slug：`aheapi`。
2. 在后台 `系统设置 -> 数据看板 -> Uptime Kuma` 添加分类：
   - 分类名称：`服务状态`
   - Uptime Kuma 地址：`https://status.aheapi.com`
   - 状态页面 Slug：`aheapi`
3. 确认 `/api/uptime/status` 能返回监控数据后，再启用控制台面板。

## 用户增长与体验指标看板计划

### 背景

- 目前缺少一组稳定、可长期对比的核心指标，难以判断用户规模、使用深度和体验变化。
- 仅看请求量不够，必须同时看活跃、消耗、留存、转化和稳定性。

### 目标

- 让管理后台能快速回答三个问题：
  1. 今天有多少人在用？
  2. 用了多少、花了多少？
  3. 用户体验是变好了还是变差了？

### 建议核心指标

#### 1. 用户规模

- DAU（日活）
- WAU / MAU
- 新增用户数
- 回访用户数
- 付费活跃用户数

#### 2. 使用深度

- 日请求数
- 日消耗金额 / 日消耗额度
- 人均请求数
- 人均消耗金额
- Top 模型消耗占比
- Top 渠道消耗占比

#### 3. 留存与粘性

- 次日留存
- 7 日留存
- 30 日留存
- 新用户首日转化率
- 首次使用后再次下单/再次调用率

#### 4. 体验与稳定性

- 请求成功率
- 5xx / 4xx 占比
- 平均首字节时间
- 平均首响应时间
- 流式请求中断率
- 客户端主动取消率

#### 5. 商业化指标

- 新增充值金额
- 订阅开通数
- 续费率
- ARPU / ARPPU
- 余额消耗速度

### 图表建议

1. 折线图：DAU、日请求数、日消耗、成功率、首响应时间。
2. 柱状图：按模型、渠道、用户组的消耗分布。
3. Cohort 图：次日 / 7 日 / 30 日留存。
4. 漏斗图：访问 -> 登录 -> 首次调用 -> 充值/订阅 -> 复用。
5. Top 榜：高消耗用户、高消耗渠道、高失败请求来源。

### 说明

- 这些指标比单纯“请求总量”更能代表用户体验和用户数量变化。
- 如果只能先做一批，优先顺序建议是：
  1. DAU + 日消耗 + 日请求数
  2. 次日 / 7 日留存
  3. 请求成功率 + 首响应时间
  4. 充值与订阅转化

## 模型上架计划

### 背景

- 后续需要补齐更多有明确需求的模型能力，提升站内可用性和模型覆盖面。
- 优先考虑用户直接会问、会用、会付费的模型类型。

### 目标

- 增加模型选择的覆盖面。
- 提升站内对多模态、创作类、复杂推理类需求的承接能力。
- 让用户尽量在站内完成主要任务，而不是外跳到别的平台。

### 上架顺序

1. Open 生图模型
2. Claude 系列模型
3. 之后再评估其他高需求模型或专用模型

### Open 生图模型

- 先确认上游接口形态、计费方式、限流策略和返回格式。
- 评估是否需要单独的图片生成入口、任务队列和结果展示页。
- 确认前端是否需要增加图片上传、风格预设、尺寸选择和结果下载能力。
- 上线前先做小流量验证，重点看失败率、生成时延和单次成本。

### Claude 系列模型

- 补齐 Claude 常用文本/代码能力模型的接入。
- 核对消息格式、长上下文、流式响应和工具调用兼容性。
- 对照现有渠道倍率和分组策略，确认计费、额度和可用模型映射。
- 上线前先验证典型场景：
  - 对话
  - 编程辅助
  - 长上下文总结
  - 流式输出

### 风险点

- 生图模型的单次成本和响应时间通常更高，需要控制默认入口和限流。
- Claude 系列若上下文长、流式多，容易暴露渠道稳定性和排队问题。
- 新模型上架前必须确认前端展示、模型分组和计费链路都已打通。

### 验收标准

- 用户可在前台正常选择并调用新模型。
- 请求、计费、日志、额度回收链路正常。
- 至少完成一次小流量实测并确认无明显异常。

## 套餐上架恢复计划

### 背景

- 之前暂停或下架的套餐需要恢复上架，重新开放给用户购买和查看。
- 恢复上架不能只改前台展示，还要同步核对价格、权益、支付和订单状态。

### 目标

- 恢复套餐在前台和后台的正常展示与购买。
- 确保套餐信息、价格、权益、支付回调和订阅状态一致。
- 避免旧配置残留导致误售或展示错误。

### 恢复步骤

1. 核对当前套餐配置、价格、周期、权益和适用范围。
2. 检查前端入口、后台管理、支付链路和订阅/续费链路。
3. 先在测试或小流量环境验证购买、开通、回调、续费和过期流程。
4. 确认无异常后再恢复正式上架。
5. 上线后观察订单、支付成功率、退款和客服反馈。

### 风险点

- 价格或权益配置不一致，导致用户实际可用额度与页面不符。
- 旧套餐缓存或旧文案未清理，造成展示错误。
- 支付回调或订阅状态同步延迟，出现已购买但未生效的情况。

### 验收标准

- 用户可在前台正常看到并购买套餐。
- 后台可正常管理套餐状态、价格和权益。
- 订单、支付、订阅和额度发放链路正常。
- 上线后未出现明显投诉或异常订单。

## 全量请求对话内容备份计划

### 背景

- 当前生产数据库主要保存请求消费日志、token 数、模型、渠道、耗时、额度等元数据。
- 当前 `logs.content` 记录的是计费说明或错误说明，不是完整用户 prompt / assistant response。
- 后续如果要备份所有请求的对话内容，需要单独设计存储链路，避免把主库快速撑大。

### 当前容量估算

- 生产服务器当前剩余磁盘约 `79G`。
- 按 2026-05-04 的生产流量估算，完整对话纯文本约 `8.1GB/天`。
- 如果以 JSON 明文保存，可能达到 `9-12GB/天`。
- 如果按天压缩为 `jsonl.zst` 或 `jsonl.gz`，预计可能降到 `1-4GB/天`，具体取决于重复内容和请求结构。
- 不建议长期把完整对话内容无压缩写入 PostgreSQL 主库，否则高峰流量下可能 1-2 周内吃满磁盘。

### 推荐方案

1. 请求对话内容按天写入压缩归档文件，例如 `request-backups/YYYY-MM-DD/*.jsonl.zst`。
2. PostgreSQL 只保存索引元数据：
   - `request_id`
   - `user_id`
   - `token_id`
   - `model_name`
   - `created_at`
   - `prompt_tokens`
   - `completion_tokens`
   - `archive_path`
   - `archive_offset` 或对象 key
   - `payload_size`
3. 本地磁盘只保留短期数据，例如 7 天或 14 天。
4. 长期备份转存到对象存储、独立备份盘或异地服务器。
5. 对敏感字段做脱敏或加密，至少避免明文保存 API key、Authorization header、渠道 key、支付信息。
6. 备份链路必须异步执行，不能阻塞 `/v1/*` 请求响应。

### 实现待办

1. 明确备份范围：
   - 是否只备份文本对话请求。
   - 是否备份响应内容。
   - 是否包含失败请求、流式中断请求、图片/音频/视频任务。
2. 设计归档格式：
   - 每行一个 JSON 对象。
   - 包含 request 元信息、请求体、响应体、错误信息、耗时和 token 使用量。
   - 大字段尽量压缩存储，避免重复保存可从日志表恢复的元数据。
3. 增加异步写入队列：
   - 请求完成后投递备份任务。
   - 后台 worker 批量写入当天归档文件。
   - worker 异常时记录错误日志，但不影响用户请求。
4. 增加本地落盘保护：
   - 单日归档文件滚动。
   - 磁盘剩余空间低于阈值时自动暂停备份。
   - 写入失败时告警，不让业务请求失败。
5. 增加保留期清理：
   - 本地归档超过保留天数自动删除。
   - 删除前确认已成功上传到长期备份位置。
6. 增加长期备份上传：
   - 优先评估 S3 兼容对象存储或独立备份服务器。
   - 上传成功后记录校验和、大小和归档日期。
7. 增加查询/恢复工具：
   - 按 `request_id` 定位归档文件和偏移。
   - 支持按日期、用户、模型导出一小段样本。
   - 恢复工具必须默认脱敏展示。
8. 增加配置开关：
   - 默认关闭全量对话备份。
   - 支持按用户组、模型、令牌或比例采样开启。
   - 支持只备份 request、不备份 response 的低成本模式。

### 风险点

- 对话内容可能包含隐私、密钥、商业代码或用户敏感数据，必须谨慎控制访问权限。
- 流式响应如果逐 chunk 记录，写入量和实现复杂度都会明显增加。
- 如果备份链路同步写入或队列无上限，可能拖慢 API 请求或造成内存上涨。
- 如果没有清理策略，完整对话备份比当前消费日志增长快两个数量级。

### 验收标准

1. 开启备份后，正常请求延迟无明显增加。
2. 归档文件能按天生成并压缩。
3. `request_id` 能准确定位到对应请求和响应内容。
4. 本地保留期清理正常，不会无限占用服务器磁盘。
5. 磁盘低水位保护能自动暂停备份并产生告警。
6. 备份内容不包含 Authorization、渠道密钥、支付密钥等高风险明文。
