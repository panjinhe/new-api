# 计划清单

## 服务器迁移后收尾计划

### 背景

- 生产服务已从阿里云迁移到新服务器。
- 当前主站与正式 API 地址为 `https://aheapi.com/v1`。
- `pbroe.com/v1` 暂时保留兼容，避免旧客户端配置立即失效。

### 待办事项

1. 观察 1-2 天，看用户调用、订单、余额、渠道日志是否稳定。
2. 之后决定是否关闭 `pbroe.com/v1` 兼容。
3. 确认没问题后，再考虑释放或降配阿里云，避免继续付固定服务器费用。
4. 定期备份新服务器 PostgreSQL。

## 无中断部署改造计划

### 目标

- 将当前“单实例原地替换”的发布方式，升级为“尽量不中断用户请求”的发布流程。
- 对普通短请求做到无感切换。
- 对流式请求 / SSE / 长连接做到尽量不被强断。

### 当前现状

- 当前部署脚本是 `deploy.sh`，核心方式是 `docker compose up -d --build --remove-orphans`。
- 当前生产 Compose 只有一个 `new-api` 服务实例，对外固定绑定 `127.0.0.1:3000:3000`。
- 当前应用入口已改为 `http.Server` 启动，并监听 `SIGTERM` / `SIGINT` 做优雅停机。
- 现状下发布新版本时，会先替换旧容器，因此存在短暂中断窗口。

### 结论

- 以当前架构，无法做到严格无中断发布。
- 要实现无中断或接近无中断，建议改为“蓝绿部署”方案。

### 推荐方案

- 保留 Nginx 作为入口层。
- 同时维护两套应用实例，例如：
  - `new-api-blue`
  - `new-api-green`
- 新版本先在未承载流量的实例上启动并完成健康检查。
- 健康检查通过后，Nginx upstream 切流到新实例。
- 观察无异常后，再下线旧实例。

### 必做改造项

#### 1. 应用优雅停机已完成

- 当前应用已支持 `http.Server` 启动。
- 当前应用已监听 `SIGTERM` / `SIGINT`。
- 当前收到停止信号后会调用 `Shutdown(ctx)` 等待存量请求完成。
- 后续只需继续校准流式请求的等待时间上限。

#### 2. 增加双实例部署配置

- 新增蓝绿部署专用 Compose 配置。
- 两个实例分别监听不同本地端口，例如：
  - blue: `127.0.0.1:3001`
  - green: `127.0.0.1:3002`
- 两个实例共用同一数据库与必要挂载目录。

#### 3. Nginx 支持动态切流

- 增加 upstream 配置，支持在 blue / green 之间切换。
- 发布时仅 reload Nginx，不重启 Nginx。
- 切流前必须先验证目标实例健康状态。

#### 4. 发布脚本升级

- 新增 `deploy-zero-downtime.sh`。
- 发布流程建议为：
  1. 构建新镜像
  2. 启动空闲颜色实例
  3. 等待健康检查通过
  4. 切换 Nginx upstream
  5. 观察日志和健康状态
  6. 下线旧实例

#### 5. 数据库迁移规范

- 所有迁移必须优先保证前后版本兼容。
- 禁止在切流阶段执行破坏性迁移。
- 推荐采用“两阶段迁移”：
  - 第一阶段：先加字段 / 加索引 / 兼容旧逻辑
  - 第二阶段：待旧版本完全下线后，再清理旧结构

### 风险点

- 流式请求在旧实例下线时，仍可能因为等待时间不足而中断。
- 如果新旧版本共享状态但数据结构不兼容，蓝绿部署也会失败。
- 如果 Nginx 切流逻辑或健康检查条件不严谨，可能把流量切到未完全就绪的新实例。

### 建议实施顺序

1. 先做应用优雅停机。
2. 再做双实例 Compose 配置。
3. 再做 Nginx upstream 切换模板。
4. 最后实现零停机发布脚本并做完整演练。

### 验收标准

- 发布过程中首页和 API 健康检查持续可访问。
- 普通 API 请求在发布期间不出现明显 5xx 峰值。
- 流式请求中断率显著低于当前方案。
- 可以稳定完成至少 3 次连续蓝绿切换演练。

## 前端构建告警与性能优化计划

### 背景

- 当前 `vite build` 可成功通过，但存在 3 类告警：
  - Vite chunk 体积偏大
  - `lottie-web` 含 `eval`
  - Browserslist 数据较旧
- 这些问题目前都不是发布阻塞项，不需要因为它们中断正常发版。

### ROI 判断

- `Vite chunk 体积偏大`：中高 ROI，值得后续排期处理。
- `lottie-web` `eval` 告警：低 ROI，暂不单独立项。
- Browserslist 数据较旧：低 ROI，适合并入常规前端依赖升级。

### 当前判断依据

- 已完成控制台主要页面的路由级懒加载，`Token`、`Log`、`Setting`、`Playground`、`Channel` 等页面不再直接进入主入口包。
- 最新构建中主入口包已从约 6.86MB 降至约 4.55MB，gzip 后从约 1.61MB 降至约 0.97MB。
- 构建仍有大 chunk 警告，主要集中在 `Semi UI`、Markdown/Mermaid/KaTeX 渲染链，以及一个共享业务 chunk。
- `lottie-web` 不是业务代码直接引入，而是通过 `@douyinfe/semi-ui` 依赖链带入。

### 建议目标

- 不追求“清空所有 warning”，优先做对用户体验真正有收益的体积优化。
- 重点降低首屏和控制台首次进入时的下载体积。
- 将低价值告警并入后续依赖维护，而不是单独花时间清理。

### 推荐后续动作

1. 暂不继续拆大 chunk，除非真实用户加载瀑布图或生产访问数据证明它仍是主要瓶颈。
2. 后续如继续优化，优先对重依赖模块做按需延迟加载，关注：
   - Markdown / Mermaid / KaTeX 渲染链
   - Dashboard 图表相关依赖
   - `react-fireworks` 这类低频特效依赖
3. 对共享业务 chunk 先做构成分析，再决定是否拆分，避免为了消除 warning 增加过多请求和维护复杂度。
4. 每轮优化后重新执行 `bun run build`，记录主包和首屏相关 chunk 体积变化。
5. 在下一次前端依赖升级时顺手更新 Browserslist 数据。
6. `lottie-web` 仅在以下场景再单独处理：
   - 需要收紧 CSP
   - 安全扫描要求清理 `eval`
   - 上游 `Semi UI` 提供更干净的替代版本

### 暂定优先级

- Done：首屏 / 路由拆包优化
- P2：真实加载瀑布图分析后，再决定是否继续拆 Markdown/Mermaid 和共享业务 chunk
- P3：Browserslist 例行更新
- P3：`lottie-web` 告警跟踪，不主动投入专项时间

## Uptime Kuma 监控面板 ROI 计划

### 当前决策

- 暂时禁用控制台「服务可用性」面板。
- 原因是当前没有接入 Uptime Kuma 公开状态页，空卡片会降低用户观感。
- 该面板只用于展示监控状态，不影响 API 调用、渠道调度、计费或令牌功能。

### ROI 判断

- 未配置监控时展示空卡片：负 ROI，容易让用户误以为站点未维护。
- 在用户量较小时立即搭建完整监控体系：中低 ROI，维护成本和误报成本偏高。
- 有付费用户或客服咨询量上来后，接入少量核心监控：中高 ROI，可减少「是不是平台挂了」这类重复咨询。

### 低成本推荐方案

后续如需启用，优先使用独立环境部署 Uptime Kuma，并只监控 2-4 个核心公开地址：

1. `https://aheapi.com/`
2. `https://aheapi.com/api/status`
3. `https://aheapi.com/v1/models`
4. `https://aheapi.com/logo-001.jpg`（可选）

建议监控间隔设置为 60-120 秒，失败重试 2-3 次，避免网络抖动造成误报。

### 后续启用条件

- 用户开始频繁询问服务是否可用。
- 有稳定付费用户，需要提高信任感和降低售后成本。
- 已有独立机器或外部环境可部署 Uptime Kuma，避免和主站同机宕机导致监控一起失效。

### 后续配置方式

1. 在 Uptime Kuma 中创建公开 Status Page，例如 slug：`aheapi`。
2. 在后台 `系统设置 -> 数据看板 -> Uptime Kuma` 添加分类：
   - 分类名称：`服务状态`
   - Uptime Kuma 地址：`https://status.aheapi.com`
   - 状态页面 Slug：`aheapi`
3. 确认 `/api/uptime/status` 能返回监控数据后，再启用控制台面板。

## 用户增长与体验指标看板计划

### 背景

- 目前缺少一组稳定、可长期对比的核心指标，难以判断用户规模、使用深度和体验变化。
- 仅看请求量不够，必须同时看活跃、消耗、留存、转化和稳定性。

### 目标

- 让管理后台能快速回答三个问题：
  1. 今天有多少人在用？
  2. 用了多少、花了多少？
  3. 用户体验是变好了还是变差了？

### 建议核心指标

#### 1. 用户规模

- DAU（日活）
- WAU / MAU
- 新增用户数
- 回访用户数
- 付费活跃用户数

#### 2. 使用深度

- 日请求数
- 日消耗金额 / 日消耗额度
- 人均请求数
- 人均消耗金额
- Top 模型消耗占比
- Top 渠道消耗占比

#### 3. 留存与粘性

- 次日留存
- 7 日留存
- 30 日留存
- 新用户首日转化率
- 首次使用后再次下单/再次调用率

#### 4. 体验与稳定性

- 请求成功率
- 5xx / 4xx 占比
- 平均首字节时间
- 平均首响应时间
- 流式请求中断率
- 客户端主动取消率

#### 5. 商业化指标

- 新增充值金额
- 订阅开通数
- 续费率
- ARPU / ARPPU
- 余额消耗速度

### 图表建议

1. 折线图：DAU、日请求数、日消耗、成功率、首响应时间。
2. 柱状图：按模型、渠道、用户组的消耗分布。
3. Cohort 图：次日 / 7 日 / 30 日留存。
4. 漏斗图：访问 -> 登录 -> 首次调用 -> 充值/订阅 -> 复用。
5. Top 榜：高消耗用户、高消耗渠道、高失败请求来源。

### 说明

- 这些指标比单纯“请求总量”更能代表用户体验和用户数量变化。
- 如果只能先做一批，优先顺序建议是：
  1. DAU + 日消耗 + 日请求数
  2. 次日 / 7 日留存
  3. 请求成功率 + 首响应时间
  4. 充值与订阅转化

## 套餐上架恢复计划

### 背景

- 之前暂停或下架的套餐需要恢复上架，重新开放给用户购买和查看。
- 恢复上架不能只改前台展示，还要同步核对价格、权益、支付和订单状态。

### 目标

- 恢复套餐在前台和后台的正常展示与购买。
- 确保套餐信息、价格、权益、支付回调和订阅状态一致。
- 避免旧配置残留导致误售或展示错误。

### 恢复步骤

1. 核对当前套餐配置、价格、周期、权益和适用范围。
2. 检查前端入口、后台管理、支付链路和订阅/续费链路。
3. 先在测试或小流量环境验证购买、开通、回调、续费和过期流程。
4. 确认无异常后再恢复正式上架。
5. 上线后观察订单、支付成功率、退款和客服反馈。

### 风险点

- 价格或权益配置不一致，导致用户实际可用额度与页面不符。
- 旧套餐缓存或旧文案未清理，造成展示错误。
- 支付回调或订阅状态同步延迟，出现已购买但未生效的情况。

### 验收标准

- 用户可在前台正常看到并购买套餐。
- 后台可正常管理套餐状态、价格和权益。
- 订单、支付、订阅和额度发放链路正常。
- 上线后未出现明显投诉或异常订单。
